SynACK encryption format

I. RSA_BLOCK structure

There is a structure named RSA_BLOCK which is used in key block encryption and file encryption.
This structure has to be encrypted with RSA 2048 bit:

// RSA_BLOCK: structure to be encrypted with RSA
#pragma pack(push, 1)
typedef union _RSA_BLOCK 
{
   struct  
   {
       ULONG   dwMagic;                        // magic (signature)
       BYTE    AesKey[32];                     // AES key
       DWORD   dwSize;                         // size of encrypted data
   };
   CHAR        Padding[RSA_BLOCK_LENGTH / 2];  // padding
}
RSA_BLOCK, *LPRSA_BLOCK;
#pragma pack(pop)

Generally this structure is filled with random values to get random AES-256 key and random padding.
Next, the dwMagic field is set to verify the correctness of decryption. The current value is 0xFA3EC9D8.
After all the dwSize is set to identify how many bytes were encrypted.
As for the file encryption, dwSize shows how many bytes are encrypted starting from the beginning of the file.
As for the key block encryption, dwSize shows how many preceeding data bytes are encrypted with AES-256.

II. Key block creation

Each SynACK encryption tool contains public master key inside it. The key is RSA-2048 public key.
At the start SynACK generates session key pair, also RSA-2048.
Then it allocates enough bytes to store the KEY_BLOCK and Path field and fills the structure:

// RSA_PRV_KEY_SIZE: size of RSA private key in bytes
#define RSA_PRV_KEY_SIZE 1412

// KEY_BLOCK: structure which stores private key and some computer info
#pragma pack(push, 1)
typedef struct _KEY_BLOCK
{
   WCHAR       wszDomainName   [DNLEN + 1];    // domain name
   WCHAR       wszComputerName [CNLEN + 1];    // computer name
   WCHAR       wszUserName     [   20 + 1];    // user name
   CHAR        szBuildID       [   20 + 1];    // build ID
   BYTE        WindowsVersion;                 // Windows version
   DWORD       ipAddress;                      // IPv4 address
   DWORD       dwTimeStamp;                    // UNIX timestamp
   DWORD       dwKeyID;                        // salt (random data)
   BYTE        PrivateKey[RSA_PRV_KEY_SIZE];   // RSA private key
   DWORD       cbPrivateKey;                   // number of bytes in private key
   CHAR        Path[1];                        // variable length startup path (UTF-8 encoded)
}
KEY_BLOCK, *LPKEY_BLOCK;
#pragma pack(pop)

Letâs explain the meaning of each field.

  • wszDomainName contains domain name if the computer is in domain.
  • wszComputerName contains NetBIOS name of the computer.
  • wszUserName stores the current user name.
  • szBuildID contains build ID for the current build.
  • WindowsVersion contains integer identifying Windows version.
  • ipAddress is set to the IPv4 address of the computer.
  • dwTimeStamp contains current Unix 32-bit timestamp.
  • dwKeyID is set to the random value and is saved as the session key ID.
  • PrivateKey contains session private RSA-2048 key in Microsoft private key BLOB format.
  • cbPrivateKey stores the length of the private key.
  • Path contains current path of the SynACK tool executable.

So, all the fields are filled.
Private session key is stored in PrivateKey and has to be wiped from RAM. The public session key is kept in RAM.

Itâs time to encrypt this block. At first itâs size should be aligned up to AES_BLOCK_SIZE which is 16.
Next, RSA_BLOCK is created and filled with random key. The dwSize is set to non-aligned size of the KEY_BLOCK.
Next, KEY_BLOCK is encrypted with RSA_BLOCK->AesKey using AES-256 in ECB mode.
Then RSA_BLOCK is encrypted with RSA-2048 using public master key.
After all encrypted KEY_BLOCK and encrypted RSA_BLOCK are concatenated together and encoded with base64.
This block has to be put inside each README file.

III. File encryption

During the previous step at the beginning SynACK generated session key pair.
The private key was stored in KEY_BLOCK, encrypted and wiped from RAM.
The public key is used to encrypt files.

To encrypt file, first of all SynACK determines the count of bytes to encrypt.
It depends on the file size and extension and there is no need to dig into itâs implementation.
Everything we should know is that this size must be aligned down to AES_BLOCK_SIZE which is 16.

Next, we create RSA_BLOCK structure and fill it with random bytes, also we set dwMagic and dwSize values.
Then we encrypt dwSize bytes from the beginning of the file using random AesKey in RSA_BLOCK with AES-256.
And now itâs time to fill FILE_BLOCK structure and put it at the end of the file after all data:

// RSA_BLOCK_LENGTH: RSA encrypted block length
#define RSA_BLOCK_LENGTH 256

// FILE_BLOCK: structure with file ECC block, hash and signatures
#pragma pack(push, 1)
typedef struct _FILE_BLOCK
{
   DWORD       dwMagic1;                       // first magic number
   DWORD       dwMagic2;                       // second magic number
   DWORD       dwKeyID;                        // key ID
   DWORD       dwBlockSize;                    // size of encrypted block
   WCHAR       wszFileName[MAX_PATH + 1];      // original file name
   BYTE        Block[RSA_BLOCK_LENGTH];        // RSA block
}
FILE_BLOCK, *LPFILE_BLOCK;
#pragma pack(pop)
  • dwMagic1 is used to verify that the file is encrypted with SynACK, it is set to 0xAE871C35.
  • dwMagic2 is used to verify that the file is encrypted with SynACK, it is set to 0x8FEC21D9.
  • dwKeyID is the key ID which was generated on the previous step in KEY_BLOCK.
  • dwBlockSize is the size of RSA encrypted data in Block field.
  • wszFileName keeps the original file name.
  • Block is the RSA_BLOCK structure encrypted with RSA 2048 bit using public session key.

So, the SynACK tool fills all the fields of FILE_BLOCK structure.
Then it encrypts RSA_BLOCK structure for this file and puts it into the Block field of FILE_BLOCK.
The size of RSA_BLOCK encrypted is stored in dwBlockSize.

Finally, FILE_BLOCK is written at the end of file after all data and file is renamed with random extension.

IV. Steps do decrypt key block

So, all files on the computer are encrypted and README files are saved in some directories.
We have to extract clientâs session key from base64 block which is inside the README file.
Here are the exact steps:

  1. We take base64 data from the README file and decode it.
  2. We take RSA_BLOCK_LENGTH bytes from the end of the data and decrypt them with master key.
  3. If the decryption was successful, we have RSA_BLOCK decrypted.
  4. We should check RSA_BLOCK->dwMagic to see if decryption was correct.
  5. We align up RSA_BLOCK->dwSize to AES_BLOCK_SIZE which is 16 and decrypt bytes from the beginning with AesKey.
  6. The result is KEY_BLOCK, where we can read information about Windows and import PrivateKey required for decryption.
  7. Also itâs necessary to save KEY_BLOCK->dwKeyID.

IV. Steps do decrypt file

OK, we have session private key.
Letâs decrypt file:

  1. First of all we read sizeof(FILE_BLOCK) bytes from the end of the file.
  2. Then we check FILE_BLOCK->dwMagic1 and FILE_BLOCK->dwMagic2 to check if this file is really encrypted with SynACK.
  3. Then we check FILE_BLOCK->dwKeyID to check if we have the correct key with matching KEY_BLOCK->dwKeyID.
  4. If everything is correct we decrypt FILE_BLOCK->Block with RSA 2048 bit using session private key.
  5. The result is RSA_BLOCK; we should check RSA_BLOCK->dwMagic to see if decryption was correct.
  6. Decrypting RSA_BLOCK->dwSize bytes from the beginning of the file with RSA_BLOCK->AesKey.
  7. Finally the file is truncated to remove FILE_BLOCK from the end and renamed to FILE_BLOCK->wszFileName.
